# TACIT Benchmark v0.1.0 -- Evaluation Guide

This document describes how to evaluate models against the TACIT Benchmark. It covers both evaluation tracks, the expected input/output formats, the verification pipeline, and a complete end-to-end worked example.

**Audience:** Researchers and engineers evaluating vision models or multimodal LLMs on TACIT.

## 1. Overview

TACIT evaluates models through two complementary tracks on the same puzzle instances:

| Track | Input to Model | Output from Model | Evaluation Method |
|-------|---------------|-------------------|-------------------|
| Track 1 (Generative) | Puzzle image (PNG) | Solution PNG image | CV-based verification via `generator.verify()` |
| Track 2 (Discriminative) | Puzzle image + N candidate images | Integer index (0-based) | Exact index match |

Both tracks operate on puzzles generated by the same deterministic generators. Track 1 tests whether a model can *produce* a correct solution. Track 2 tests whether a model can *recognize* the correct solution among near-miss distractors.

The performance gap between Track 1 and Track 2 on the same model and the same puzzles is itself a research signal -- it measures the difference between generative and discriminative visual reasoning capacity.

## 2. Track 1 -- Generative Evaluation

### 2.1 What the Model Receives

The model receives a single puzzle image as a PNG. The image encodes all information necessary to determine the solution. No natural language prompt or textual clue is provided beyond what is visually encoded in the image itself.

For example, a maze puzzle image shows the grid with walls, the start marker (green "S"), the end marker (red "E"), portal indicators (colored circles), and layer labels.

### 2.2 What the Model Must Produce

The model must produce a **solution PNG image** that visually encodes the structural answer. The task-specific verifier uses computer vision (CV) techniques to extract the answer from the rasterized image and check it against the ground truth.

Each task defines what visual information the verifier expects. The key point: **verification is CV-based, operating on pixel data.** The verifier analyzes the PNG to extract the candidate answer, then checks it against the ground truth. Different tasks use different CV strategies (pixel sampling, color counting, SSIM comparison, BFS path tracing).

#### Example: Maze Solution PNG

A correct maze solution PNG shows the maze grid with the solution path drawn in blue (#2266FF). The maze verifier (`MazeGenerator.verify()`) detects blue path pixels, maps them to grid cells, and traces the connected component via BFS. It then checks:

1. The path starts at the maze's designated start cell.
2. The path ends at the maze's designated end cell.
3. Every cell in the path is a passage (no wall breaches).
4. Every pair of consecutive cells is either adjacent on the same layer or connected by a portal.

#### Example: Raven's Matrices Solution PNG

A correct Raven's solution PNG renders the missing tile with the correct shape, color, size, rotation, and count attributes. The Raven verifier (`RavenGenerator.verify()`) computes the Structural Similarity Index (SSIM) between the candidate PNG and the ground truth tile PNG. A match requires SSIM >= 0.997.

### 2.3 How Verification Works

Verification is a three-step pipeline:

```
Model's PNG output
    |
    v
[1] Visual extraction (CV analysis of PNG to extract candidate data)
    |
    v
[2] Constraint checking (validate extracted data against puzzle rules)
    |
    v
[3] VerificationResult(passed=bool, reason=str, details=dict)
```

Each generator implements its own `verify(puzzle: PuzzleInstance, candidate_png: bytes) -> VerificationResult` method. The `VerificationResult` contains:

- `passed` (bool): Whether the candidate is a correct solution.
- `reason` (str): Human-readable explanation of the result.
- `details` (dict): Task-specific diagnostic information (e.g., number of mismatched cells, SSIM score, list of violated constraints).

The evaluation harness calls this method for each puzzle-candidate pair. See `tacit/evaluation/track1.py`:

```python
def evaluate_generative(
    generator: BaseGenerator,
    puzzle: PuzzleInstance,
    candidate_png: bytes,
) -> VerificationResult:
    return generator.verify(puzzle, candidate_png)
```

### 2.5 Verification Contracts by Task

Each generator defines its own verification logic. The structural elements each verifier looks for are summarized below. For complete details, see [Task Specifications](task-specifications.md).

| Task | CV Verification Strategy | What It Checks |
|------|--------------------------|----------------|
| Maze | Detects blue path pixels (#2266FF), maps to grid cells, traces connected component via BFS | Start/end correctness, passage-only cells, adjacency/portal connectivity |
| Raven | SSIM comparison against ground truth tile PNG (threshold 0.997) | Visual similarity to expected solution tile |
| CA Forward | Pixel sampling at grid cell centers, color-to-state mapping | Cell-by-cell comparison against deterministic simulation result |
| CA Inverse | Pixel sampling of rule table cells, color-to-state mapping | Entry-by-entry comparison of rule table arrays |
| Logic Grid | Symbol color sampling at cell centers with inverse-distance weighted voting | Latin square validity, all constraint satisfaction |
| Graph Coloring | Pixel sampling at node center positions with occlusion-aware offsets | No adjacent conflicts, exactly k colors used |
| Graph Isomorphism | Green vs red pixel count for binary answer detection | Matches ground truth boolean |
| Unknot | Green vs red pixel count for binary answer detection | Matches ground truth classification |
| Ortho Projection | Filled vs empty cell pixel sampling against known grid layout | Cell-by-cell match against deterministic projection |
| Iso Reconstruction | SSIM comparison against ground truth PNG (threshold 0.99999) | Visual similarity to expected isometric view |

## 3. Track 2 -- Discriminative Evaluation

### 3.1 What the Model Receives

The model receives:

1. **Puzzle image** (PNG): The same puzzle image as in Track 1.
2. **N candidate solution images** (PNGs): One correct solution plus N-1 distractor images.

The candidates are presented as an indexed list. The correct solution is placed at a random position among the distractors.

### 3.2 What the Model Must Produce

The model must output a single **0-based integer index** identifying which candidate it believes is the correct solution.

### 3.3 Expected JSON Format

Track 2 model outputs must be provided as a JSON file with the following structure:

```json
{
  "results": [
    {
      "puzzle_id": "maze_easy_0042",
      "task": "maze",
      "difficulty": "easy",
      "correct_index": 2,
      "selected_index": 2
    },
    {
      "puzzle_id": "maze_easy_0043",
      "task": "maze",
      "difficulty": "easy",
      "correct_index": 0,
      "selected_index": 3
    },
    {
      "puzzle_id": "raven_easy_0042",
      "task": "raven",
      "difficulty": "easy",
      "correct_index": 1,
      "selected_index": 1
    }
  ]
}
```

**Field definitions:**

| Field | Type | Description |
|-------|------|-------------|
| `puzzle_id` | string | Unique puzzle identifier (matches `PuzzleInstance.puzzle_id`) |
| `task` | string | Task name (one of the 10 task identifiers) |
| `difficulty` | string | Difficulty level (`easy`, `medium`, `hard`) |
| `correct_index` | int | 0-based index of the correct solution in the candidate array |
| `selected_index` | int | 0-based index the model selected |

Evaluation is a simple exact-match comparison: `correct_index == selected_index`.

### 3.4 The Distractor System

Distractors are **not random images**. They are near-miss solutions generated by the same task generator, each violating exactly one structural constraint. This design prevents Track 2 from collapsing into trivial pattern matching.

Each task defines specific violation types. The default configuration generates 4 distractors per puzzle, cycling through the available violation types. With 1 correct solution and 4 distractors, the random baseline is 20%.

**Distractor violation types by task:**

| Task | Violation Types |
|------|----------------|
| Maze | `wall_breach` (path goes through a wall), `portal_skip` (layer transition without portal), `disconnected` (non-adjacent consecutive cells), `wrong_exit` (path ends at wrong cell) |
| Raven | `wrong_shape`, `wrong_color`, `wrong_rotation`, `wrong_count` |
| CA Forward | `wrong_cell` (flipped cells), `wrong_step_count` (simulated wrong number of steps), `wrong_rule` (applied a different rule) |
| CA Inverse | `off_by_one_rule` (one entry changed), `transposed_rule` (two entries swapped), `partial_rule` (multiple entries changed) |
| Logic Grid | `constraint_violation` (violates one constraint), `symbol_swap` (two symbols swapped in a row), `non_unique` (different valid Latin square) |
| Graph Coloring | `adjacent_conflict` (two adjacent nodes same color), `missing_color` (fewer than k colors), `wrong_k` (more than k colors) |
| Graph Isomorphism | `opposite_answer` (binary -- the wrong boolean) |
| Unknot | `opposite_answer` (binary -- the wrong classification) |
| Ortho Projection | `wrong_axis` (projected along different axis), `missing_feature` (removed cells), `extra_feature` (added phantom cells), `mirrored` (left-right flip) |
| Iso Reconstruction | `wrong_depth` (modified depth), `missing_face` (removed voxels), `extra_volume` (added phantom voxels), `rotated` (90-degree rotation) |

## 4. End-to-End Example

This section walks through a complete evaluation cycle from puzzle generation to results.

### Step 1: Generate Puzzles

Generate 5 maze puzzles at easy difficulty, starting from seed 42:

```bash
tacit generate --task maze --difficulty easy --count 5 --seed 42 --output-dir data
```

### Step 2: Examine Generated Files

The generator produces the following structure under the output directory:

```
data/
├── svg/
│   └── maze/
│       └── easy/
│           ├── maze_easy_0042_puzzle.svg
│           ├── maze_easy_0042_solution.svg
│           ├── maze_easy_0042_distractor_0.svg
│           ├── maze_easy_0042_distractor_1.svg
│           ├── maze_easy_0042_distractor_2.svg
│           ├── maze_easy_0042_distractor_3.svg
│           ├── maze_easy_0043_puzzle.svg
│           ├── maze_easy_0043_solution.svg
│           ├── ... (distractors for 0043)
│           ├── maze_easy_0044_puzzle.svg
│           ├── ... (through seed 0046)
├── png/
│   └── maze/
│       └── easy/
│           ├── maze_easy_0042_puzzle_512.png
│           ├── maze_easy_0042_solution_512.png
│           ├── maze_easy_0042_distractor_0_512.png
│           └── ...
└── metadata/
    └── maze/
        └── easy/
            ├── maze_easy_0042.json
            ├── maze_easy_0043.json
            └── ...
```

Each metadata JSON file contains the full `PuzzleInstance` data:

```json
{
  "task": "maze",
  "puzzle_id": "maze_easy_0042",
  "seed": 42,
  "difficulty": {
    "level": "easy",
    "params": {"grid_size": 8, "layers": 1, "portals": 0}
  },
  "puzzle_svg": "data/svg/maze/easy/maze_easy_0042_puzzle.svg",
  "solution_svg": "data/svg/maze/easy/maze_easy_0042_solution.svg",
  "distractor_svgs": [
    "data/svg/maze/easy/maze_easy_0042_distractor_0.svg",
    "data/svg/maze/easy/maze_easy_0042_distractor_1.svg",
    "data/svg/maze/easy/maze_easy_0042_distractor_2.svg",
    "data/svg/maze/easy/maze_easy_0042_distractor_3.svg"
  ],
  "distractor_violations": [
    "wall_breach",
    "portal_skip",
    "disconnected",
    "wrong_exit"
  ]
}
```

### Step 3: What the Model Receives

**Track 1 input:** The model is shown `maze_easy_0042_puzzle_512.png`. It must produce a solution PNG.

The puzzle image shows an 8x8 maze grid on a single layer with a green "S" marker at the start and a red "E" marker at the end. All walls, passages, and the grid border are visible.

**Track 2 input:** The model is shown `maze_easy_0042_puzzle_512.png` along with 5 candidate solution images (1 correct + 4 distractors), each rendered as a PNG. The model must output the 0-based index of the correct candidate.

### Step 4: What the Model Must Output

**Track 1 output:** A PNG file at `model_output/maze/easy/maze_easy_0042.png` showing the maze with the solution path drawn in blue (#2266FF). The verifier detects blue path pixels, maps them to grid cells, and traces connectivity via BFS. The path must form a connected, wall-respecting route from start to end.

**Track 2 output:** A JSON results file:

```json
{
  "results": [
    {
      "puzzle_id": "maze_easy_0042",
      "task": "maze",
      "difficulty": "easy",
      "correct_index": 2,
      "selected_index": 2
    }
  ]
}
```

### Step 5: Run Evaluation

**Track 1:**

```bash
tacit evaluate --track generative --model-output ./model_output --tasks maze --output results_t1.json
```

**Track 2:**

```bash
tacit evaluate --track discriminative --model-output ./track2_results.json --tasks maze --output results_t2.json
```

The evaluation harness produces a results report:

## 5. Evaluation CLI Reference

### Generate Puzzles

```bash
# Single task, single difficulty
tacit generate --task maze --difficulty easy --count 100 --seed 42

# Full suite from config
tacit generate --config configs/default.yaml

# Custom output directory and distractor count
tacit generate --task raven --difficulty hard --count 50 --seed 0 \
    --output-dir ./my_data --distractors 8
```

### Evaluate Model Outputs

```bash
# Track 1: Generative
tacit evaluate --track generative \
    --model-output ./model_output \
    --tasks all \
    --output results.json

# Track 2: Discriminative
tacit evaluate --track discriminative \
    --model-output ./discriminative_results.json \
    --tasks all \
    --output results.json

# Evaluate specific tasks only
tacit evaluate --track generative \
    --model-output ./model_output \
    --tasks maze,raven,ca_forward \
    --output results.json
```

### Publish to HuggingFace

```bash
tacit publish --config configs/full_release.yaml \
    --hf-repo tylerxdurden/TACIT-benchmark
```

## 6. Cross-Track Analysis

When a model is evaluated on both tracks using the same puzzle set, the gap between Track 1 and Track 2 performance reveals the model's generative-versus-discriminative reasoning profile.

| Scenario | Track 1 | Track 2 | Interpretation |
|----------|---------|---------|---------------|
| Strong generative reasoning | High | High | Model understands the problem deeply |
| Recognition without generation | Low | High | Model can identify but cannot produce solutions |
| Generation without recognition | High | Low | Unusual; may indicate task-specific shortcuts |
| Weak reasoning | Low | Low | Model lacks the required reasoning capability |

This cross-track signal is available per task and per difficulty level, enabling fine-grained capability profiling.
